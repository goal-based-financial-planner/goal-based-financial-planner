## Phase 4: User Story 2 — Compare Actual vs Suggested (Priority: P2)

**Goal**: The Track section shows a per-row comparison of suggested monthly amount vs actual logged amount for the selected month, with the difference highlighted.

**Independent Test**: Log one entry for a goal in the current month → open Track section → verify the comparison table row for that investment type shows the correct suggested amount, the logged amount, and their difference. Verify rows with no logged amount show "₹0" actual and a negative difference.

- [x] T011 [P] [US2] Implement `buildMonthlyComparison(entries: InvestmentLogEntry[], suggestions: InvestmentSuggestion[], month: number, year: number): MonthlyInvestmentSummary[]` as a named export in `src/domain/investmentLog.ts`; for each suggestion, sum all entries matching `investmentName + month + year`; set `actualAmount` to the sum (or 0); set `difference = actualAmount - suggestedAmount` (depends on T001)
- [x] T012 [US2] Extend `src/pages/Planner/components/GoalCard/components/InvestmentTracker/index.tsx`: (a) add a month/year DatePicker navigator (using `@mui/x-date-pickers`, `views={['month','year']}`) defaulting to the current month; (b) call `buildMonthlyComparison` with the selected month and render one table row per result showing investment name, suggested amount (₹), actual amount (₹), and difference (₹, coloured red for shortfall / green for surplus or zero) (depends on T011)
- [x] T013 [US2] Add empty state handling in `src/pages/Planner/components/GoalCard/components/InvestmentTracker/index.tsx`: when no entries exist for the selected month, all rows show ₹0 actual with negative difference equal to the suggested amount; the "Log Investment" button remains prominent

**Checkpoint**: US2 fully functional. Comparison table visible in Track section. Month can be changed via picker. Rows correctly reflect sum of logged amounts vs suggestions.

## Phase 6: User Story 4 — Cumulative Investment Progress (Priority: P4)

**Goal**: A cumulative totals row at the bottom of the Track section shows the total actual invested across all months vs the total that should have been invested by now (current allocations × elapsed months).

**Independent Test**: Log entries across several months for a goal → verify the cumulative actual total in Track section equals the sum of all logged amounts. Verify the cumulative suggested total equals the sum of all suggestion amounts multiplied by the goal's elapsed months. Confirm both values update when a new entry is logged.

- [x] T019 [P] [US4] Implement two named exports in `src/domain/investmentLog.ts`: (a) `calculateCumulativeActual(entries: InvestmentLogEntry[]): number` — sum of all entry amounts; (b) `calculateCumulativeSuggested(suggestions: InvestmentSuggestion[], elapsedMonths: number): number` — sum of all suggestion amounts multiplied by `elapsedMonths` (depends on T001)
- [x] T020 [US4] Add a cumulative totals section to `src/pages/Planner/components/GoalCard/components/InvestmentTracker/index.tsx`: render a summary row showing "Total Invested: ₹X" (from `calculateCumulativeActual`) and "Should Have Invested: ₹Y" (from `calculateCumulativeSuggested` using `goal.getElapsedMonths()`) with the shortfall/surplus highlighted. Render this above the month navigator and comparison table (depends on T019)

**Checkpoint**: US4 fully functional. Cumulative totals appear in Track section. Totals update when any entry is added, edited, or deleted.

## Notes

- [P] tasks = different files, no shared state, can run concurrently
- [Story] label maps each task to its user story for traceability
- `crypto.randomUUID()` requires an evergreen browser — no polyfill needed for this project's target audience
- The `PlannerData` constructor signature change (adding `investmentLogs` as a third optional param) may cause TypeScript errors in existing test files that call `new PlannerData(goals, allocations)` — these will compile without changes since the param is optional
- Run `npm run build` after Phase 2 before starting any user story phase to catch type errors early
- Commit after each phase checkpoint to keep history clean and reversible
